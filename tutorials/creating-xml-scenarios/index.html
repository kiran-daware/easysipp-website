Creating XML Scenarios

Master the visual XML builder and create complex SIPp scenarios with ease

Introduction

One of easySIPp's most powerful features is its visual XML scenario builder. Instead of manually writing SIPp XML files, you can create complex call flows through an intuitive point-and-click interface.

Understanding SIPp XML Scenarios

SIPp XML scenarios define the sequence of SIP messages exchanged during a test. A typical scenario includes:

  • INVITE: Initiates a call
  • 100 Trying: Provisional response
  • 180 Ringing: Call is ringing
  • 200 OK: Call accepted
  • ACK: Acknowledgment
  • BYE: Terminates the call

Note: easySIPp includes predefined scenarios for common use cases, but you can create custom scenarios for specific testing needs.

Accessing the XML Builder

There are two ways to access the visual XML builder:

Method 1: From easySIPp Interface

  1. Click on the "Xml List" tab in the navigation
  2. Browse available scenarios or click "Create Xml" to start a new one

Method 2: Standalone XML Generator

Visit the online tool at: https://kiran-daware.github.io/sipp-xml/

This standalone version allows you to create scenarios and export them for use in easySIPp.

Creating Your First Custom Scenario

Step 1: Define the Scenario Type

Choose whether you're creating a:

  • UAC Scenario: Initiates calls (client behavior)
  • UAS Scenario: Receives calls (server behavior)

Step 2: Add SIP Messages

Build your call flow by adding messages in sequence:

For UAC:

  1. Click "Add Message" ‚Üí Select "send"
  2. Choose message type: INVITE
  3. Configure headers (From, To, Contact, etc.)
  4. Add SDP body if needed

For UAS:

  1. Click "Add Message" ‚Üí Select "recv"
  2. Choose message type: INVITE
  3. Define expected headers (optional)

Step 3: Add Responses

After each request, add appropriate responses:

Step 4: Set Timing and Pauses

Add pauses between messages to simulate realistic behavior:

  • milliseconds: Quick responses (100-500ms)
  • seconds: Call duration (10-30s typical)

Using Variables in Scenarios

Variables make scenarios reusable and dynamic:

Common Variables:

  • [service] - Called party number
  • [field0] - Caller party number
  • [remote_ip] - Remote IP address
  • [remote_port] - Remote port
  • [local_ip] - Local IP address
  • [local_port] - Local port

Example INVITE with Variables:

INVITE sip:[service]@[remote_ip]:[remote_port] SIP/2.0
Via: SIP/2.0/UDP [local_ip]:[local_port]
From: 
To: 
Contact: 
Call-ID: [call_id]
CSeq: 1 INVITE

Building Common Scenarios

Basic UAC Call Scenario

A simple outbound call:

  1. Send INVITE

  3. Receive 180 Ringing (optional)
  4. Receive 200 OK
  5. Send ACK
  6. Pause (call duration)
  7. Send BYE

  4. Pause (ring duration)
  5. Send 200 OK
  6. Receive ACK
  7. Pause (call duration)

Exporting and Using Your Scenario

From easySIPp Interface:

  1. Click "Save" after building your scenario
  2. Give it a descriptive name
  3. It will appear in the XML dropdown for UAC/UAS

From Standalone Generator:

  1. Click "Export XML"
  2. Save the file (e.g., my_scenario.xml)
  3. Upload to easySIPp via the interface

Testing Your Scenario

Step 1: Preview the call flow:

  • Select your custom scenario
  • Click "Preview Combined XML Call Flow"
  • Verify the message sequence is correct

Step 2: Run a small test:

  • Set Total Calls to 1-5
  • Set CPS to 1
  • Monitor for errors

Step 3: Scale up:

  • Once verified, increase call volume
  • Adjust CPS as needed

Best Practices

  • Start simple: Begin with basic scenarios before adding complexity
  • Use variables: Make scenarios reusable across different environments
  • Add comments: Document complex logic in XML comments
  • Test incrementally: Verify each message addition works
  • Follow SIP standards: Ensure proper SIP message formatting
  • Include error handling: Plan for unexpected responses

Common Mistakes to Avoid

  • Missing ACK: Always send ACK after receiving 200 OK to INVITE
  • Incorrect CSeq: CSeq must increment for each new request
  • Missing Contact header: Required for proper routing
  • Wrong message order: Follow proper SIP call flow sequences
  • Hardcoded IPs: Use variables instead for flexibility

Troubleshooting

Issue: Scenario fails immediately

Solution: Check SIPp output for XML parsing errors. Verify:

  • All tags are properly closed
  • Mandatory headers are present
  • Message sequence is logical

Issue: Calls timeout

Solution: Ensure:

  • Expected responses match what's actually sent
  • Timing/pauses are realistic
  • Network connectivity is working
