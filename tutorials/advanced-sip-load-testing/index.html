Advanced Load Testing

Master high-volume testing, performance tuning, and stress testing techniques

Introduction

Once you're comfortable with basic SIP testing, it's time to explore advanced load testing capabilities. This tutorial covers high call rates, performance optimization, distributed testing, and capacity planning.

Understanding Load Testing Metrics

Key Performance Indicators

  • CPS (Calls Per Second): Rate at which new calls are initiated
  • Concurrent Calls: Number of active calls at any given moment
  • Success Rate: Percentage of completed calls vs. attempted
  • Response Time: Time between request and response
  • Call Duration: Average length of completed calls
  • Failure Rate: Percentage and types of failures

Calculating Capacity

# Example: 10 CPS √ó 60 seconds = 600 concurrent calls
#
# System capacity = Max CPS before failure rate exceeds threshold

Planning Your Load Test

Define Test Objectives

Before starting, clearly define what you're testing:

  • Baseline Performance: Normal operating capacity
  • Peak Load: Maximum expected traffic
  • Stress Test: Breaking point identification
  • Endurance Test: Sustained load over time
  • Spike Test: Sudden traffic increases

Test Environment Setup

Critical: Always test in a non-production environment that mirrors production as closely as possible.

Ensure you have:

  • Adequate hardware resources (CPU, RAM, network)
  • Isolated network to avoid impacting production
  • Monitoring tools in place
  • Clear understanding of expected results

Progressive Load Testing

Step 1: Baseline Test

Start with a low call rate to establish baseline:

Total Calls: 100
CPS: 1
Duration: ~100 seconds

Expected Result: 100% success rate

Step 2: Incremental Increase

Gradually increase load:

Test 1: CPS 1   ‚Üí Monitor
Test 2: CPS 5   ‚Üí Monitor
Test 3: CPS 10  ‚Üí Monitor
Test 4: CPS 25  ‚Üí Monitor
Test 5: CPS 50  ‚Üí Monitor

Step 3: Find the Breaking Point

Continue increasing until you see:

  • Success rate drops below 95%
  • Response times increase significantly
  • System errors or timeouts
  • Resource exhaustion (CPU, memory, network)

Managing System Resources

CPU Optimization

  • Use UDP instead of TCP (less overhead)
  • Minimize logging during high-load tests
  • Close unnecessary applications
  • Use multiple cores if available

Memory Management

# Monitor memory usage during tests
docker stats easysipp

# For manual installations
top -p $(pgrep -f sipp)

Network Tuning

# Increase network buffer sizes (Linux)
sudo sysctl -w net.core.rmem_max=26214400
sudo sysctl -w net.core.wmem_max=26214400

# Increase file descriptors
ulimit -n 65536

Distributed Load Testing

Multi-Machine Setup

When a single machine can't generate enough load:

  1. Deploy multiple easySIPp instances
    • Machine 1: UAC 1 (CPS 50)
    • Machine 2: UAC 2 (CPS 50)
    • Machine 3: UAC 3 (CPS 50)
    • Total: 150 CPS
  2. Coordinate test execution
    • Start all UACs simultaneously
    • Use synchronized clocks
    • Aggregate results manually

Performance Optimization

Scenario Optimization

  • Minimize pauses: Remove unnecessary delays in XML
  • Reduce SDP size: Only include required codecs
  • Simplify call flows: Skip optional provisional responses
  • Use variables efficiently: Pre-calculate values when possible

Protocol Selection

Protocol Pros Cons
UDP Lower overhead, higher CPS No reliability, packet loss
TCP Reliable, ordered delivery More CPU, lower CPS

Recommendation: Use UDP for maximum performance, TCP when reliability is critical.

Real-Time Monitoring During Load Tests

Key Metrics to Watch

  • Success Rate: Should stay above 95%
  • Failed Calls: Note failure reasons
  • Response Times: Should remain consistent
  • System Resources: CPU, memory, network usage
  • Target System: Monitor the SUT (System Under Test)

Using easySIPp's Control Screen

During a running test, you can:

  • Increase CPS: Click "+" to add more load
  • Decrease CPS: Click "-" to reduce load
  • Pause: Temporarily stop new calls
  • Stop: End the test immediately

Pro Tip: Use real-time rate adjustment to find the exact breaking point without running multiple tests.

Stress Testing Patterns

Ramp-Up Test

Minutes 0-5:   CPS 10  (warm-up)
Minutes 5-10:  CPS 25
Minutes 10-15: CPS 50
Minutes 15-20: CPS 100
Minutes 20-25: CPS 150 (find breaking point)

Spike Test

Minutes 0-10:  CPS 10  (baseline)
Minutes 10-12: CPS 200 (spike)
Minutes 12-22: CPS 10  (recovery)
Minutes 22-24: CPS 200 (second spike)

Endurance Test

# Run at 80% capacity for extended period
CPS: 80 (if max is 100)
Duration: 4-8 hours
Monitor: Memory leaks, connection leaks, performance degradation

Analyzing Results

Success Criteria

  • ‚úÖ Pass: 95%+ success rate at target load
  • ‚ö†Ô∏è Warning: 90-95% success rate (needs investigation)
  • ‚ùå Fail: Below 90% success rate

Common Bottlenecks

  • CPU Bound: High CPU usage ‚Üí Optimize scenarios, use UDP
  • Memory Bound: High memory ‚Üí Reduce concurrent calls
  • Network Bound: Packet loss ‚Üí Check bandwidth, MTU size
  • Application Bound: Target system can't handle load

Best Practices

  • Start small: Always begin with low rates
  • Monitor everything: Both load generator and target
  • Test incrementally: Don't jump from 10 to 1000 CPS
  • Document results: Keep records of all tests
  • Replicate production: Match real-world scenarios
  • Plan for failure: Know your rollback strategy
  • Test regularly: Capacity changes over time

Troubleshooting High-Load Issues

Issue: Can't achieve target CPS

Solutions:

  • Check system resources (CPU, RAM, network)
  • Simplify XML scenarios
  • Use UDP instead of TCP
  • Distribute load across multiple machines

Issue: High failure rate at moderate load

Solutions:

  • Check target system capacity
  • Verify network connectivity
  • Review SIPp error messages
  • Increase timeout values

Issue: System becomes unresponsive

Solutions:

  • Reduce concurrent calls
  • Increase system resources
  • Check for memory leaks
  • Monitor file descriptor limits
